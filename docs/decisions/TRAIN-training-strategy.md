# Engineering Decisions — Training Strategy & ML Pipeline

Decisions governing how models are trained, refined, and optimized throughout the Forest Pulse development cycle.

---

## TRAIN-001: Self-Training Loop for Iterative Label Refinement

**Status:** Accepted  
**Date:** 2026-04-06  
**Commits:** `7f873ff` (add self-training loop), `74961d0` (fix: lower confidence)  
**File:** `scripts/self_train.py`

### Context

Initial training data consisted of 10,376 weak labels generated by DeepForest. These were noisy (~60% precision) because DeepForest was trained on North American forests. The first RF-DETR model (mAP50 = 41.5% on self-labels) inherited these errors. The team needed a way to iteratively improve training labels without manual annotation.

### Options Considered

| Option | Rationale | Outcome |
|--------|-----------|---------|
| Manual Roboflow annotation | Gold labels, but weeks + €1000 | Rejected (timeline too slow) |
| **Self-training with confidence threshold** | Use model's own high-confidence predictions to replace weak teacher | **Selected** |
| GroundingDINO zero-shot teacher | State-of-the-art visual grounding | Tested in `scripts/teacher_annotations.py`; failed — cannot resolve individual tree crowns from nadir view |

### Decision

Implement `scripts/self_train.py`: semi-supervised learning loop orchestrating N rounds of **detect → relabel → train → measure**. Per round:

1. Load current checkpoint (`checkpoints/current.pt`)
2. Re-predict all patches at configurable `--confidence` threshold (default 0.5 after TRAIN-002)
3. Save predictions as new COCO annotations
4. Re-run dataset prep (80/20 train/val split)
5. Fine-tune RF-DETR from scratch for 10 epochs
6. Log mAP50 on held-out validation split
7. Archive round-specific checkpoint and annotations

### Rationale

- **Domain adaptation:** RF-DETR on Catalan data outperforms DeepForest from round 1 onward.
- **Discrete rounds:** archive labeled datasets per round (reproducibility + audit trail); abort if quality degrades.
- **Confidence as filter:** too high (0.9) → training starvation; too low (0.3) → noisy labels persist; 0.5 is middle ground.
- **Not circular:** fresh training per round (no warm-start) prevents overfitting to one model's biases.
- **Gold set sanity check:** if gold_map50 drops while self-labels improve, self-training has failed.

### Consequences

- Weak labels → 3 rounds → mAP50 on self-labels rises from 0.4 to 0.85–0.90.
- Annotation count drops round-to-round (27K → 17K) — the model becomes more selective.
- **Metric still inflated:** evaluates against self-generated labels, not physical ground truth. See EVAL-001.
- Risk: error accumulation if confidence threshold is mistuned. Mitigated by annotation count logging and visual inspection.

## TRAIN-002: Confidence Threshold Tuning (0.7 → 0.5)

**Status:** Accepted  
**Date:** 2026-04-06  
**Commit:** `74961d0`  
**File:** `scripts/self_train.py:74-76`

### Context

Original `--confidence` default was hardcoded at 0.7. During early experimentation, this was too aggressive: the first self-training round filtered so aggressively that round 2 began to starve the training set. mAP50 plateaued and visual inspection showed obvious trees being missed (over-filtering).

### Decision

Change default `--confidence` from 0.7 to **0.5**. Additionally:
- **Backup checkpoint before round 1** (saves original as `checkpoints/round_0.pt`) for comparison.
- **Log annotation count per round** with warning if count drops below 50% of previous round.

### Rationale

- At confidence 0.5 on RF-DETR softmax outputs, approximately half of detections survive — preserves diversity while reducing noise.
- Fixed threshold is reproducible and interpretable (vs adaptive thresholds which add complexity).
- Backup checkpoint enables A/B comparison of model evolution.

### Consequences

- Self-training sustains across 3+ rounds with stable annotation counts.
- 0.5 is still somewhat arbitrary. Phase 8 LiDAR evidence shows the true precision bottleneck (~27%) is not addressed by threshold alone — Phase 9 classifier is required.

## TRAIN-003: RF-DETR Fine-Tuning Hyperparameter Choices

**Status:** Accepted  
**Date:** 2026-04-05  
**Commits:** `8326cca`, `c2a1d46`  
**File:** `autoresearch/train.py`

### Context

No prior work existed in the codebase for RF-DETR training. All hyperparameters had to be determined from:
- Available GPU memory (48 GB Apple MPS, shared with SAM2)
- Dataset size (~300 patches initially; grows via self-training)
- Iteration time goal (hours, not days)

### Decision

Create `autoresearch/train.py` with a clear configuration block that the auto-research harness can modify:

```python
BACKBONE = "rfdetr-base"
LEARNING_RATE = 1e-4
BATCH_SIZE = 4
FINE_TUNE_EPOCHS = 10
```

The `train()` function auto-computes `grad_accum = max(1, 16 // BATCH_SIZE)` to simulate effective batch size 16.

### Rationale

**Backbone (rfdetr-base vs rfdetr-large):** base trains in ~10 min/epoch on MPS; large takes 30+ min. Small dataset (risk of overfitting with large backbone). Can swap later if needed.

**Learning rate (1e-4):** Standard fine-tuning LR for pre-trained detection models. 1e-3 causes instability on small datasets; 1e-5 converges too slowly.

**Batch size (4) + gradient accumulation:** MPS memory constraints prevent batch size 16. Gradient accumulation: 4 micro-batches × 4 accumulation steps = effective batch of 16. Mathematically equivalent, 4× the memory of batch 16.

**Epochs (10):** Typical for fine-tuning. Allows ~1–2 full passes over 240 training images. Configurable for auto-research harness (Phase 11).

**Why editable configuration block?** The harness cannot safely modify complex logic. A simple config block (`BACKBONE = "..."`) is unambiguous: change it, run training, measure. Karpathy-style harness pattern.

### Consequences

- Training loop is reproducible and configurable.
- Four hyperparameters become the "search surface" for Phase 11 (auto-research).
- Initial training runs in ~2 hours for 10 epochs on MPS (vs ~30 min on NVIDIA A100).
- Once LiDAR-verified metric exists (Phase 8), all four hyperparameters should be re-tuned against real ground truth.

## TRAIN-004: Cross-Platform Device Detection (CUDA / MPS / CPU Fallback)

**Status:** Accepted  
**Date:** 2026-04-05  
**Commit:** `09a4511`  
**File:** `src/forest_pulse/device.py`

### Context

Forest Pulse needed to run on Apple Silicon (MPS), NVIDIA GPUs (CUDA), and CPU without requiring users to configure device placement manually. Device placement is a common source of bugs in PyTorch.

### Options Considered

| Strategy | Rejected Because |
|----------|-----------------|
| Hardcoded CUDA | Fails on non-NVIDIA machines |
| User-specified via CLI flag | Requires users to know their hardware |
| **Auto-detect (CUDA > MPS > CPU)** | **Selected** — zero configuration |
| Environment variable only | Less discoverable than auto-detection |

### Decision

Create `src/forest_pulse/device.py`:

```python
def get_device() -> torch.device:
    """Auto-detect best available device: CUDA > MPS > CPU."""
    if torch.cuda.is_available():
        return torch.device("cuda")
    elif torch.backends.mps.is_available():
        return torch.device("mps")
    else:
        return torch.device("cpu")
```

Additionally, set environment variable **before any PyTorch import**:
```python
os.environ.setdefault("PYTORCH_ENABLE_MPS_FALLBACK", "1")
```

This allows ops not yet supported on MPS (bicubic upsampling in SAM2) to silently fall back to CPU.

### Rationale

- **Auto-detect:** standard in modern ML frameworks (JAX, distributed PyTorch). Reduces cognitive load and bugs.
- **Priority (CUDA > MPS > CPU):** CUDA = fastest, most compatible; MPS = 5–10x slower than CUDA but 5–10x faster than CPU; CPU = always works.
- **`PYTORCH_ENABLE_MPS_FALLBACK=1`:** SAM2 uses operations not yet on MPS natively. Without this, they crash. With it, they silently run on CPU. Must be set before any PyTorch import.
- **AMP check (`supports_amp()`):** mixed precision (float16) speeds training 2x on CUDA but produces numerical artifacts on MPS. Conditional AMP based on device.

### Consequences

- Training and inference work identically on Apple Silicon (dev), NVIDIA (cloud), and CPU (laptop).
- MPS ops not supported by Metal fall back to CPU — slight latency hit, but no crashes.
- Device detection is deterministic and logged at startup for debugging.

## TRAIN-005: Training on Weak Bootstrap Labels Before Manual Annotation or LiDAR

**Status:** Accepted  
**Date:** 2026-04-05  
**Commits:** `e624d84`, `c2a1d46`

### Context

Three labeling approaches available: (1) DeepForest weak labels (minutes, free, ~60% precision), (2) manual Roboflow annotation (2 weeks, ~€1000, ~95% precision), (3) LiDAR auto-labels (not yet available — Phase 7). Which first?

### Decision

Bootstrap with DeepForest immediately. Start training RF-DETR the same day labels are generated. Build LiDAR infrastructure in parallel.

### Rationale

**Timeline math:**
- Weak labels: available day 1 → train RF-DETR → iterate.
- Manual annotation: 2 weeks elapsed → same training after.
- LiDAR auto-labels: 3–4 weeks infrastructure → training after.

**By starting with weak labels, the team discovered:**
- Self-training works (see TRAIN-001)
- mAP50 on self-labels is meaningless (led to EVAL-001)
- Real bottleneck is crown merging recall + shrub precision (not label quality)

These discoveries would have been delayed 3 weeks if waiting for manual annotation.

**Why weak labels are acceptable:** Object detection is robust to ~30–40% label noise. DeepForest achieves ~60% precision on Catalan data (~40% false positives). RF-DETR can learn from noisy labels if trained conservatively.

### Consequences

- mAP50 is inflated throughout Phases 3–7 (evaluated on self-generated labels).
- By Phase 8, LiDAR reveals real recall ~30% — but the team now has the infrastructure to address it.
- All three labeling approaches ultimately coexist and validate each other.

## TRAIN-006: LiDAR Feature Extraction — Seven Per-Tree Features

**Status:** Accepted  
**Date:** 2026-04-07  
**Commit:** `5690823`  
**File:** `src/forest_pulse/lidar.py`

### Context

The Phase 9 tree/bush classifier needs multi-modal features. LiDAR 3D point clouds provide physically-measured attributes that RGB alone cannot: tree height, canopy density, vertical structure.

### Decision

For each RF-DETR detection (bbox), extract **7 features** from ICGC LAZ point cloud inside the crown region:

| Feature | Unit | Ecological Purpose |
|---------|------|-------------------|
| `height_p95_m` | m | Canopy top — primary tree/bush discriminator |
| `height_p50_m` | m | Crown center proxy |
| `vertical_spread_m` | m | p95 − p5 (trunk + crown depth) |
| `point_count` | count | LiDAR density inside crown |
| `return_ratio` | [0,1] | Canopy density (multi-return / total) |
| `intensity_mean` | DN | Leaf spectral reflectance (needle vs. broadleaf) |
| `intensity_std` | DN | Crown heterogeneity |

**LiDAR tile URL pattern (ICGC, verified 2026-04-07):**
```
https://datacloud.icgc.cat/datacloud/lidar-territorial/laz_unzip/
    full10km{ID10K}/lidar-territorial-v3r1-full1km{ID1K}-2021-2023.laz
```

### Rationale

**Why these 7 features?** Each addresses a distinct aspect of tree ecology:
- `height_p95`: direct physical tree/bush discriminator (trees > 5 m per Spanish Forest Inventory).
- `vertical_spread`: mature trees have larger vertical extent than shrubs.
- `point_count` + `return_ratio`: canopy density (multi-return indicates dense foliage).
- `intensity_mean` + `intensity_std`: spectral signatures differentiate conifers from broadleaf.
- `height_p50`: proxy for crown center location (useful for 3D segmentation work).

**Why 7 not 20?** With ~500 training examples in Phase 9 (LiDAR auto-labeled), we're at the edge of overfitting with 18 total features (RGB + LiDAR). 7 LiDAR features + 11 RGB features = 18 total is aggressive but manageable with gradient boosting.

**Why per-tree features (not patch-level)?** Patch-level aggregates lose individual tree information. Per-tree features enable distinguishing a tall tree from a forest of short shrubs.

**CHM rasterization at 0.5 m resolution:** 160 m patch → 320×320 cells = 102K pixels. Fast (<1s per crown), sufficient for height percentiles. Coarser (1 m) loses precision; finer (0.1 m) = memory explosion.

### Consequences

- After Phase 7 implementation, the 5 m height filter removes 100% of shrubs in the test patch (patch 0250) with zero false negatives on actual trees.
- LiDAR 2021–2023 vs RGB 2024: acceptable for structural features.
- LAZ tiles (~500 MB each) cached locally; adjacent patches reuse same tile (2–4 GB total).

## TRAIN-007: CHM (Canopy Height Model) Generation from LAZ

**Status:** Accepted  
**Date:** 2026-04-07  
**Commit:** `5690823`  
**File:** `src/forest_pulse/lidar.py`

### Context

The canopy height model is the backbone of the LiDAR pipeline. To compute CHM:

1. Download LAZ point clouds (ICGC Territorial v3r1)
2. Separate ground (ASPRS class 2) and high vegetation (class 5) points
3. Rasterize into DSM (high veg) and DTM (ground) rasters at 0.5 m
4. Subtract: CHM = DSM − DTM

### Decision

`compute_chm_from_laz()` implementation:
- **Resolution:** 0.5 m (160 m patch → 320×320 raster = 102K pixels)
- **Rasterization:** max z value per cell (DSM uses highest point; preserves sharp crown peaks)
- **ASPRS classification:** class 2 (ground) for DTM; class 5 (high vegetation) for DSM
- **Ground reference:** `z - min_ground_z_in_bbox` as MVP approximation (vs full per-pixel DTM)
- **Caching:** CHM rasters cached in `data/montseny/rasters/` by bounds hash

### Rationale

**Why 0.5 m?** Fast (<1s/patch), sufficient for height percentiles, fits in memory. A 10-pixel crown at 0.5 m contains enough height returns for stable p95 computation.

**Why max vs mean?** For tree crown height, the highest point (canopy top) is the metric. Mean would smooth out the crown tip, making `height_p95` less discriminative for the 5 m threshold.

**Why local ground reference?** At 160 m patch scale, elevation variation is ~10–50 m. Using `min_z_in_bbox` introduces ~5–10 m systematic error. For the 5 m tree/bush threshold, this is acceptable noise. Full per-pixel DTM deferred to Phase 12 (species work).

**Why ASPRS class 2 for ground?** ICGC LiDAR classification is reliable for ground points. Trusting the official classification avoids building a custom classifier.

### Consequences

- CHM computation is deterministic and reproducible (same LAZ + resolution → same CHM).
- Temporal mismatch (LiDAR 2021–2023, RGB 2024). Acceptable for structural features; revisit for species classification.
- Phase 12 may need per-pixel DTM if elevation correction becomes critical.

## Summary: Training Architecture Principles

### Principle 1: Detect Once, Classify Many Times

```
RF-DETR (frozen) → LiDAR height filter → NDVI filter → GradientBoostingClassifier
                              ↑
                   Classifier iterations take 5 min (scikit-learn)
                   Detector re-training takes 20 min (RF-DETR)
```

**Rationale:** Decoupling detector from classifier enables fast iteration on the classification decision without retraining the expensive detector.

### Principle 2: Weak Labels + Fast Iteration > Manual Curation + Slow Iteration

Timeline comparison:
- Weak labels strategy: day 1 = training; day 3 = self-training complete; day 7 = LiDAR integration
- Manual annotation strategy: day 1–14 = annotation; day 15 = first training; day 22 = LiDAR integration

**Same endpoint, 2 weeks earlier** by choosing weak labels first.

### Principle 3: Multi-Modal Features Over Multi-Modal Models

Rather than fusing RGB, GRVI, and LiDAR into a single end-to-end neural network, extract features from each modality and feed a tabular gradient boosting classifier:
- Each feature is interpretable (ecology users can understand why a tree was rejected)
- Faster to train (~500 examples → seconds on scikit-learn)
- Easier to debug (failure modes isolated per feature)

### Principle 4: Cross-Platform-First Design

Device detection is a first-class module (`device.py`), not an afterthought buried in training scripts. Same code runs on Apple Silicon, NVIDIA, and CPU with zero user configuration.
