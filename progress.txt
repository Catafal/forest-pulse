## Key learnings (the honest ones)
- mAP 0.904 was inflated 8.4× — real F1 is 0.108 (LiDAR-verified) at
  the original conf=0.30 default
- Real recall WAS 6.8% in dense Mediterranean canopy at conf=0.30,
  but Phase 10a showed this was a self-imposed limit: at conf=0.02,
  recall climbs to 15.3% and F1 reaches 0.252 (with 9.5a filter)
- The single biggest "leak" in 4 phases of work was the default
  inference confidence threshold, untouched since Phase 1. ALWAYS
  sweep inference hyperparameters before retraining or feature
  engineering. The 30-min Phase 10a experiment unlocked more F1
  than any classifier or feature-engineering work in Phases 9 / 9.5.
- Precision after 9.5a filter degrades gracefully with candidate
  density (0.98 → 0.72) but the filter still composes additively
  with raw recall improvements
- rfdetr top_k=300 cap is the new ceiling — Phase 10b lever
- LiDAR tree-top matching gives a deterministic filter that hits the
  ceiling. Use it directly in Catalunya. No learning required.
- RGB at 25 cm/px does NOT contain enough signal to recover the LiDAR
  precision boundary on the RF-DETR output distribution. The
  RGB-distillation hypothesis is empirically refuted on this dataset.
- NDVI can't distinguish trees from bushes (both chlorophyll-rich)
- Foundation segmentation models (SAM2) help modestly on distinct
  crowns, not on uniform canopy
- Detection and classification should still be separate concerns
  (Phase 12 species classification should reuse the Phase 9.5b
  infrastructure with species labels — different question, may yield
  different result because species cues exist at 25 cm/px)

### Phase 10b: Raise rfdetr top_k cap — TESTED, DEAD END
- [x] rfdetr architecture: pred_logits is [B, 300, 2] → 300 queries
      × 2 classes = 600 (query, class) candidates max. num_select=1000
      raises RuntimeError from torch.topk.
- [x] Modified scripts/sweep_confidence.py to support --num-select INT
      and --class-zero-only flags. Default behavior unchanged.
- [x] Ran the full sweep at num_select=600 (with and without class
      filter) — F1 IDENTICAL to Phase 10a in every row.
- [x] Reason: the trained class (0) dominates every query, so the
      top-300 of the 600-candidate pool is ALWAYS entirely class 0.
      The 300 extra slots at num_select=600 fill exclusively with
      class-1 noise from an untrained head, which (a) sits below the
      0.01 confidence floor, (b) gets dropped by the 9.5a filter.
- [x] CONFIRMED: Phase 10a's F1 = 0.252 IS the legitimate ceiling
      for this RF-DETR checkpoint on 640x640 input. There is no
      hidden recall available via the topk lever.
- [x] See .claude/runs/confidence-sweep-recall-experiment/RESULTS-phase10b.md
- [x] Lesson learned: a saturation cap is only meaningful if it's
      binding on the SIGNAL, not just on the raw count. Always check
      the class/category distribution of the topk before assuming the
      cap is the bottleneck.

## Core architectural principle

**Detect once, classify many times.**
RF-DETR finds crown-like regions. A separate classifier decides what each one is.
The detector stays frozen unless evidence says otherwise. Classification improvements
iterate in seconds (retrain XGBoost) instead of 20+ minutes (retrain RF-DETR).
